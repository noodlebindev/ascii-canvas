# ascii-canvas — Design Spec

**Date:** 2026-05-17
**Status:** Spec — ready for implementation plan
**Skill type:** Superpowers-style markdown skill (no code, no external tool dependencies)
**Skill root:** `~/Skills/ascii-canvas/`
**Output root:** `~/Ascii-diagrams/`

---

## 1. Purpose

A "Canva for ASCII." Given a topic, idea, or set of facts, compose ASCII/Unicode text-based visual artifacts — diagrams, slideshows, infographics, playbooks, cheat sheets, and 11 other formats — using a fixed catalog of templates with canonical narrative arcs, strict layout discipline, and predictable output behavior.

The skill is fundamentally a guide that teaches Claude *how* to produce high-quality ASCII compositions. It contains no executable code. All output is generated by Claude reading the skill's templates, rules, and canonical arcs.

## 2. Skill identity & triggers

### 2.1 Skill description (controls auto-triggering)

> Compose ASCII/Unicode diagrams, slideshows, infographics, playbooks, and other text-based visual artifacts from a topic, idea, or set of facts. Use when the user explicitly requests a diagram/visual/format (flowchart, slideshow, cheat sheet, infographic, timeline, etc.) **or** asks for a structured breakdown / step-by-step explanation of a non-trivial topic.
>
> Do **NOT** trigger for:
> - simple factual questions
> - short answers (less than ~3 steps or one paragraph)
> - conversational or opinion-based queries

### 2.2 Trigger examples

**Explicit format requests** — "make a flowchart", "draw the architecture", "ASCII slideshow", "infographic on X", "comparison table", "timeline of", "cheat sheet for"

**Visual framing** — "visualize X", "show me how X works", "diagram this", "illustrate"

**Structured breakdown of non-trivial topics** — "break down X", "walk me through X step by step", "explain X as a playbook", "learning ladder for X", "content map for X", "make me a one-pager on X", "turn this into slides"

### 2.3 Early-exit rule

If trigger is weak AND structure does not improve clarity over plain prose → do not activate. Default to a normal text response.

## 5. Canonical narrative arcs (multi-panel formats)

Stored in `composition-rules/narrative-arcs.md`. Eight arcs. Each format file references its arc.

### 5.1 Slideshow (default: 5–7 panels)
1. Hook / framing
2. Context / why it matters
3. Core idea
4–5. Examples / applications
6. Key takeaway
7. *Optional:* recap or next step

**Use for:** teaching concepts, content posts, explainers.

### 5.2 Playbook (default: 5–8 panels)
1. Objective
2. When to use / context
3. Prerequisites
4–N. Step-by-step execution
N+1. Success criteria
N+2. Failure modes / pitfalls

**Use for:** workflows, SOPs, AI agent systems, marketing execution.

### 5.3 Learning ladder (default: 4–6 levels)
1. Foundations
2. Core mechanics
3. Practical application
4. Intermediate depth
5. Advanced nuance
6. Expert / edge cases

**Use for:** skill progression, education.

### 5.4 Infographic (default: 4–6 blocks)
1. Headline / key claim
2–4. Supporting insights (stats, concepts, contrasts)
5. Synthesis / takeaway
6. *Optional:* action or CTA

**Use for:** high-signal summaries, social content.

### 5.5 Concept stack (default: 4–6 layers)
1. Base layer (foundation)
2–N. Supporting layers (dependencies)
N+1. Top-level outcome

**Use for:** systems thinking, dependencies, architecture of ideas.

### 5.6 Content map (default: 3–5 clusters)
1. Core idea
2–4. Angle clusters — each cluster contains hooks / posts / threads
5. *Optional:* distribution notes

**Use for:** content expansion, X/social idea trees.

### 5.7 Dashboard (default: 4–8 panels)
1. Title / context
2–N. Key metrics or sections
Final. Summary / signal

**Use for:** status views, performance summaries.

### 5.8 Step-by-step guide (default: 4–7 steps)
1. Goal
2–N. Sequential steps
Final. Outcome / check

**Use for:** simple processes (lighter than playbook).

### 5.9 Universal panel rules

- **Every panel must earn its existence.** Merge similar panels. Drop weak content.
- **Transition rule:** Panel N must logically create the need for Panel N+1. If not, structure is wrong.
- **Default panel count: 4–7.** Never exceed 9 without explicit user request.
- **Compress before expanding.**
- **Content-driven reduction:** if content does not justify the full arc → reduce panel count. Do NOT introduce filler to meet default ranges.
- **User override:** if user specifies structure ("3 slides", "5 steps") → follow user verbatim.

### 5.10 Formats and thinking modes

Formats are not just layouts — they are pre-defined ways of organizing thinking:
- Slideshow = narrative
- Playbook = execution
- Learning ladder = progression
- Infographic = compression
- Concept stack = dependency
- Content map = expansion
- Dashboard = surveillance
- Step-by-step = sequence

## 6. Character set policy

Stored in `composition-rules/box-drawing.md`.

### 6.1 Default: Unicode box-drawing

```
Light:  ┌─┐ ├┤ ┬┴ ┼ │
Heavy:  ┏━┓ ┣┫ ┳┻ ╋ ┃
Double: ╔═╗ ╠╣ ╦╩ ╬ ║
Shapes: ◆ ▲ ● ◇ ◯ ▼ ■ □
Arrows: → ← ↑ ↓ ↔ ↕ ⇒ ⇐
```

### 6.2 ASCII fallback

Use pure ASCII (`+-|`, `*`, `>`, `v`, `^`) when:
- user explicitly requests ("plain ASCII", "email-safe", "copy-paste safe")
- output is destined for logs, code blocks, plain-text email, old terminals

### 6.3 Equivalence rule

The same layout must render in both Unicode and ASCII without breaking alignment. Format files ship dual skeletons where layout differs.

### 6.4 Consistency rule

Do not mix Unicode line weights in a single artifact (e.g. `─` and `═` together). Use one weight unless intentionally differentiating (e.g. heavy borders for the outer frame, light for inner cells — and only when that distinction carries meaning).

## 7. Width budgets

Stored in `composition-rules/width-budgets.md`.

| Format class | Max width |
|---|---|
| Slideshow, step-by-step-guide, learning-ladder, playbook | 80 cols |
| Cheat-sheet, comparison-table | 90–100 cols |
| Flowchart, timeline, mind-map, infographic, decision-tree, content-map, concept-stack | 100 cols |
| Architecture-diagram, sequence-diagram | 110–120 cols |
| Dashboard | 120 cols |

### 7.1 Hard rules

- Never exceed the format's max width.
- Prefer shrinking content over wrapping awkwardly.
- Maintain alignment over verbosity.

### 7.2 Overflow handling

If content exceeds width:
1. Shorten text
2. Restructure layout
3. Split into multiple panels
4. *Never* exceed max width to force content in

## 8. Composition rules

Universal rules every format inherits:

### 8.1 Box-drawing grammar
- Consistent line weights within an artifact (see §6.4)
- Closed boxes only — no dangling line segments

### 8.2 Column-alignment grammar
- Every cell within a column has identical width
- Padding is consistent (typically 1 space inside each cell)

### 8.3 Spacing grammar
- Major section separation in saved `.md`: `\n\n---\n\n`
- Minor section separation: single blank line
- Consistent vertical rhythm across all sections in a multi-panel artifact

### 8.4 Arrow grammar
- Direction matches flow: `→` forward, `←` backward, `↓` cascade, `↑` reverse cascade, `↔` bidirectional
- Don't mix `→` and `>` for the same semantic meaning in one artifact

### 8.5 Label-placement rule
- If text doesn't fit inside a box → place label outside the box with a leader line
- Never force-cram text past box boundaries

### 8.6 Visual hierarchy
- Primary content: top or left
- Indentation signals dependency / subordination
- Grouping (proximity, shared border) signals relationship

### 8.7 Density control
- Default to compact layouts
- Only increase spacing when readability genuinely requires it

### 8.8 Simplicity rule (format choice)
When multiple formats apply:
- Choose the simplest format that preserves clarity
- Prefer single-format output over combining formats
- Only combine formats if clarity improves significantly
- If two formats are equally valid → choose the simpler one
- **v1 constraint:** do not combine formats unless explicitly requested or clearly necessary

## 9. Output behavior

### 9.1 Chat-only (no file written)

- Single-artifact outputs: flowchart, decision-tree, comparison-table, timeline, mind-map, sequence-diagram, architecture-diagram, single-section cheat-sheet
- Exploratory drafts
- One-shot quick visuals
- Outputs where the user has signaled iteration ("try again", "draft", partial artifacts)

A single-artifact output is one continuous ASCII chunk with no internal section dividers. Multi-section variants of any of these formats (e.g. a 4-section cheat-sheet, a comparison-table split into themed groups) follow §9.2.

### 9.2 Auto-save + chat preview

Saved to `~/Ascii-diagrams/` when:
- Multi-panel formats by definition: slideshow, infographic, playbook, learning-ladder, content-map, concept-stack, dashboard
- Any artifact with more than one section (e.g. a multi-section cheat-sheet, a step-by-step-guide with 4+ steps each containing prose + visual)
- Clearly reusable assets (guides, frameworks, anything the user references by name)

User can always force a save with explicit phrasing ("save this", "make a file"). User can suppress a save with ("don't save", "chat only").

### 9.3 Iteration guard

Do not auto-save during exploratory or iterative outputs. Signals:
- User says "try again", "another version", "draft this"
- User is partway through providing constraints (partial artifact)
- The skill itself is on its second QA attempt after a fail

### 9.4 File path & naming

```
~/Ascii-diagrams/YYYY-MM/YYYY-MM-DD-<topic-slug>.md
```

**Slug rules:**
- Lowercase, hyphen-separated
- 6–8 words max
- Derived from user intent — not generic ("diagram-1" is forbidden)
- **Fallback if slug is unclear:** `<format>-<short-topic>` (e.g. `playbook-cold-outbound`, `slideshow-spanish-verbs`)

### 9.5 File format

Always `.md` (markdown). Structure:

```markdown
# Title

Short description (1–2 lines)

---

## Section 1

<ASCII content inside a code block>

---

## Section 2

<ASCII content>
```

Rationale: clean on GitHub/Notion/sharing, preserves formatting, future-proof for any UI built on top.

### 9.6 Chat preview rules

- Output ≤ 80 lines → render full output inline
- Output > 80 lines → render first 1–2 sections inline, indicate saved, offer "expand to show full"

### 9.7 Post-save line (exact format)

```
Saved to:
~/Ascii-diagrams/2026-05/2026-05-17-spanish-verb-conjugation.md
```

No fluff. Just the path.

### 9.8 Anti-pattern (locked)

Do not default to saving everything "just in case." If it's not a reusable artifact, it doesn't deserve a file. The cost of noise + duplicates outweighs the rare "I wish I'd saved that."

### 9.9 No partial save rule

Do not save incomplete or truncated outputs. Only fully generated artifacts may be written to file.

Specifically forbidden:
- Saving a preview render (when chat preview shows only the first 1–2 sections per §9.6, the file write must contain the full artifact, not the preview)
- Saving an artifact that failed strict QA
- Saving during a fail-twice → simplify retry mid-cycle
- Saving when the generation was interrupted

Pre-write gate: full artifact assembled + strict QA passed → then write.

## 10. QA discipline

Stored in `composition-rules/qa-checklist.md`.

### 10.1 Light QA — single in-chat diagrams

Claude eyeballs:
- Alignment looks right
- Width fits the format budget
- Boxes appear closed

### 10.2 Strict QA — saved/multi-panel/reusable/publishable artifacts

Before output, verify:
1. Every box closes
2. Every column aligns
3. No line exceeds the format width budget
4. No awkward mid-cell wrapping
5. Arrows/connectors are readable
6. Unicode and ASCII fallback structures remain layout-equivalent
7. Section spacing is consistent
8. Markdown code fences preserve formatting

### 10.3 When QA runs

- **Strict QA runs before both chat render AND file write.**
- **Do not write files that have not passed strict QA.**
- If QA passes but readability is still poor → simplify anyway.

### 10.4 Failure behavior

- Silently fix issues. Do not mention minor corrections.
- Only warn the user if the requested layout cannot fit cleanly at the chosen width.

### 10.5 Fail-twice → simplify rule

If layout fails strict QA twice:
- Reduce text
- Split into more panels
- Switch to a narrower/simpler format
- Move labels outside boxes
- Never force dense text into tight boxes

### 10.6 Format fallback (graceful degradation)

If the selected format still cannot produce a clear structure after the simplify rule fires:
- Fall back to a simpler adjacent format

Adjacency map:
- `playbook` → `step-by-step-guide`
- `slideshow` → `infographic` (collapse narrative into block summary)
- `infographic` → `cheat-sheet` (collapse blocks into flat reference)
- `dashboard` → `cheat-sheet`
- `architecture-diagram` → `concept-stack` (drop topology, keep layers)
- `sequence-diagram` → `step-by-step-guide` (drop actors, keep order)
- `mind-map` → `content-map`
- `concept-stack` → `comparison-table` (rows = layers)
- `learning-ladder` → `step-by-step-guide`
- `content-map` → `infographic`
- `flowchart` → `step-by-step-guide` (collapse branches into the dominant path; note alternatives in prose)
- `decision-tree` → `comparison-table` (rows = outcomes, columns = conditions)

When fallback fires, emit a one-line note: `Note: simplified from <original> to <fallback> for readability.`

### 10.7 Cross-artifact consistency check

In addition to the 8-point checklist, strict QA verifies that repeated structural elements stay visually consistent across the entire artifact:
- All boxes in the artifact use the same line weight (per §6.4)
- All panels in a multi-panel artifact use identical width
- All section dividers are identical (no mix of `---` and `===`)
- All bullet styles within a single artifact match (no mix of `•` and `-`)
- All arrows pointing the same direction use the same glyph

This catches subtle drift that point-checks on individual sections might miss.

## 11. Routing logic (SKILL.md entry-point flow)

```
1. DETECT TRIGGER
   ├── Explicit format keyword?        → route to that format's file
   ├── Visual framing ("visualize")?   → infer best format from topic shape
   ├── Structured breakdown of
   │   non-trivial topic?              → infer multi-panel format
   └── None of the above (or trigger
       is weak and structure won't
       improve clarity)                → DO NOT activate; respond normally

2. CHOOSE FORMAT
   If user named one → use it.
   Else apply routing matrix (topic shape → format):

   branching + decisions         → decision-tree
   branching + mixed steps       → flowchart
   linear process                → step-by-step-guide
   repeatable workflow / SOP     → playbook
   skill progression             → learning-ladder
   one idea → many outputs       → content-map
   layers / dependencies         → concept-stack
   side-by-side options          → comparison-table
   actors + messages             → sequence-diagram
   system components             → architecture-diagram
   events over time              → timeline
   central idea + radial         → mind-map
   dense reference               → cheat-sheet
   status / metrics              → dashboard
   narrative explainer           → slideshow
   high-signal summary           → infographic

   Tie-breakers:
   - When multiple formats apply → choose simplest that preserves clarity
   - When two are equally valid  → choose the simpler one
   - Do not combine formats in v1 unless explicitly requested

3. LOAD FORMAT FILE (mandatory)
   → ~/Skills/ascii-canvas/formats/<format>.md
   No ad-hoc layouts. Templates only.

4. DETERMINE CHARACTER SET
   Unicode by default. ASCII fallback if requested OR context demands it.

5. APPLY CANONICAL ARC
   Multi-panel formats only. From narrative-arcs.md.
   Reduce panel count if content doesn't justify the full arc.

6. GENERATE OUTPUT

7. DETERMINE SAVE BEHAVIOR
   Apply §9 rules. Apply iteration guard (§9.3).

8. RUN QA
   Strict mode for saved/multi-panel/reusable; light otherwise.
   Strict QA runs before render AND before file write.
   If fails twice → invoke simplify rule.

9. RENDER + (optional) SAVE
   Apply chat preview rules (§9.6).
   If saved, emit exact post-save line (§9.7).
```

## 12. Out of scope — v1

Explicitly deferred to v2 or later:

- **`mode: quick | publish` flag** — manual override of save/QA behavior. Deferred.
- **External tool integration** (graph-easy, ditaa, figlet) — documented as manual workflow only; not invoked by the skill.
- **Color / ANSI escapes** — markdown renderers strip; saved `.md` looks broken in Notion/GitHub.
- **Polish mode** — user supplies pre-built ASCII; skill cleans and aligns it. Separate feature.
- **Image export** (PNG/SVG) — requires external tools (ditaa).
- **Multi-language skill prompts** — content can be any language; skill prompts stay English.
- **Format-combination outputs** — e.g. slideshow with embedded flowchart panels. Defer until v1 produces a clear case for it.
- **A "list supported formats" command** — defer until needed.

## 14. Success criteria

The skill is v1-complete when:

1. All 38 files exist and pass internal review.
2. SKILL.md's routing logic correctly dispatches all 16 trigger categories without ambiguity.
3. Each format file produces an artifact that passes strict QA at its target width.
4. Multi-panel formats produce canonical-arc-conformant outputs without filler.
5. Auto-save behavior writes to the correct `~/Ascii-diagrams/YYYY-MM/YYYY-MM-DD-slug.md` path with valid slug naming.
6. Chat-only outputs never write a file.
7. Unicode + ASCII fallback variants produce layout-equivalent results.
8. The skill correctly refuses to activate on simple factual queries and short answers.
